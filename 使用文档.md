# 饮食情况追踪与 AI 分析建议 — 使用文档

本文档说明如何安装、配置和日常使用本系统（含 Web 前端、后端 API 与智能餐盒对接）。

---

## 一、软件简介

本系统用于**饮食情况追踪与 AI 分析建议**（模块 5），主要功能包括：

| 功能模块       | 说明 |
|----------------|------|
| **数据汇总**   | 每日 0 点自动汇总前一天的用餐记录到日汇总表，供图表与报告使用。 |
| **统计图表**   | 按用户、日期范围查看：日趋势、周对比、浪费率、用餐速度、营养饼图。 |
| **AI 报告**    | 按日或按周生成饮食分析报告（支持 OpenAI，未配置时使用规则引擎降级）。 |
| **个性化建议** | 基于最近报告与诊断规则，给出优先级分类的饮食改进建议。 |
| **设备管理**   | 智能餐盒设备与用户的绑定、解绑；硬件通过遥测接口上报数据并自动写入用餐记录。 |

---

## 二、环境要求

- **Node.js**：建议 18 及以上
- **MySQL**：5.7 或 8.x，用于存储用餐记录、日汇总、AI 报告、设备绑定等
- **浏览器**：使用前端时建议使用 Chrome、Edge、Safari 等现代浏览器

## 三、安装与配置

### 3.1 克隆/进入项目目录

```bash
cd /path/to/mit
```

### 3.2 后端依赖与数据库

1. **安装后端依赖**

   ```bash
   npm install
   ```

2. **准备 MySQL**

   - 启动 MySQL 服务（如 `mysql.server start` 或 Docker）。
   - 创建数据库：`CREATE DATABASE diet_tracking;`
   - 若需单独用户：创建用户并授权该库。

3. **配置环境变量**

   ```bash
   cp .env.example .env
   ```

   编辑 `.env`，按实际环境填写：

   | 配置项 | 说明 | 必填 |
   |--------|------|------|
   | `DB_HOST` | MySQL 主机，本地一般为 `localhost` | 是 |
   | `DB_PORT` | 端口，默认 `3306` | 是 |
   | `DB_USER` | 数据库用户名 | 是 |
   | `DB_PASSWORD` | 数据库密码 | 是 |
   | `DB_NAME` | 数据库名，如 `diet_tracking` | 是 |
   | `PORT` | 后端服务端口，默认 `3000` | 否 |
   | `CRON_DAILY_SUMMARY_ENABLED` | 是否启用每日 0 点汇总，`1` 启用 `0` 关闭 | 否，默认 1 |
   | `OPENAI_API_KEY` | OpenAI API Key，不填则用规则引擎报告 | 否 |
   | `OPENAI_MODEL` | 模型，如 `gpt-4o-mini` | 否 |

4. **执行数据库迁移**

   ```bash
   npm run migrate
   ```

   会创建/更新：`Meal_Records`、`Daily_Diet_Summary`、`AI_Analysis_Reports`、设备绑定与餐盒相关表等。

### 3.3 前端依赖与配置

1. **安装前端依赖**

   ```bash
   cd frontend
   npm install
   ```

2. **前端环境（可选）**

   - 开发时默认通过 Vite 代理把 `/api` 和 `/health` 转到后端 `http://localhost:3000`，一般无需改配置。
   - 若需指定后端地址，可在 `frontend` 目录下建 `.env`，设置：
     - `VITE_API_BASE=http://localhost:3000`（或你的后端地址）

## 四、启动服务

### 4.1 启动后端（必选）

在**项目根目录** `mit/` 下：

```bash
npm start
```

服务默认运行在 **http://localhost:3000**。可访问 http://localhost:3000/health 检查是否正常。

### 4.2 启动每日汇总定时任务（可选）

若希望每天 0 点自动汇总前一天数据，在项目根目录另开终端执行：

```bash
npm run cron
```

或通过 `.env` 中 `CRON_DAILY_SUMMARY_ENABLED=1` 与你的进程管理方式配合使用。

### 4.3 启动前端（使用 Web 界面时）

在 **frontend** 目录下：

```bash
cd frontend
npm run dev
```

浏览器访问 **http://localhost:5173**。请确保后端已启动，否则图表、报告、建议、设备等接口会请求失败。

## 五、前端使用说明

前端顶部导航包含：**首页**、**统计图表**、**AI 报告**、**个性化建议**、**设备管理**。

### 5.1 首页

- 展示系统简介与各功能入口链接，点击可进入对应页面。

### 5.2 统计图表

- **用户 ID**：选择要查看的用户（如 `1`）。
- **开始日期 / 结束日期**：查询的日期范围（YYYY-MM-DD）。
- 下方为五类图表，每类有独立 **「加载」** 按钮：
  - **日趋势**：选定日期范围内的每日指标趋势。
  - **周对比**：周与周之间的对比（柱状图）。
  - **浪费率分析**：浪费率随时间变化。
  - **用餐速度分析**：用餐速度相关指标。
  - **营养摄入（卡路里占比）**：营养/卡路里占比饼图。
- 修改用户或日期后，需在对应图表上再次点击「加载」以刷新数据。

### 5.3 AI 报告

- **用户 ID**：要生成/查看报告的用户。
- **日期**：日报为单日；周报时该日期为**周结束日**（含该日往前 7 天）。
- **类型**：**日报** 或 **周报**。
- **生成报告**：调用后端生成该日/该周的 AI 报告（若未配置 OpenAI，则使用规则引擎报告）。
- **查询已有**：仅从数据库拉取已存在的报告并展示。
- 报告内容会显示在下方（文本或结构化评价、改进措施、下周目标、分数等）。

### 5.4 个性化建议

- **用户 ID**：要查看建议的用户。
- 点击 **「加载建议」** 后，下方会列出基于最近报告与诊断规则生成的建议条目。
- 每条建议带有优先级标识（高/中/低），部分条目会有具体改进建议子项。

### 5.5 设备管理（智能餐盒）

- **查询已绑定设备**：输入 **用户 ID**，点击「查询已绑定设备」，可看到该用户下已绑定的设备 ID 列表。
- **绑定设备**：在「设备 ID」输入框中填写设备标识（如 `aa:bb:cc`），点击「绑定」。绑定成功后列表会刷新。
- **解绑**：在已绑定设备列表中，点击某设备旁的「解绑」，确认后即可解除该设备与当前用户的绑定。

## 六、智能餐盒对接（硬件/方案 A）

若需对接智能餐盒硬件，除前端「设备管理」外，还需了解以下接口与流程。

### 6.1 设备绑定关系

- 设备须先与用户绑定，遥测上报时后端根据 `device_id` 解析出 `user_id`，再写入该用户的用餐数据。
- 绑定方式：
  - **方式一**：在前端「设备管理」页绑定（见 5.5）。
  - **方式二**：调用 API：
    - 绑定：`POST /api/v1/devices/bindings`，Body：`{"device_id":"aa:bb:cc","user_id":1}`
    - 解绑：`DELETE /api/v1/devices/bindings/{device_id}`
    - 查询某用户绑定列表：`GET /api/v1/devices/bindings?user_id=1`

### 6.2 遥测上报

- 硬件按约定格式向 **`POST /api/v1/hardware/telemetry`** 上报数据。
- Body 示例：`{"device_id":"aa:bb:cc","timestamp":1715000000,"weights":{"grid_1":25,"grid_2":25,"grid_3":25,"grid_4":25}}`
- 后端根据重量与时间序列做状态机判断（IDLE → SERVING → EATING → IDLE），在合适时机写入 `Lunchbox_Meals`、`Meal_Curve_Data`，并在结算时同步到 `Meal_Records`。

### 6.3 测试 7 拍完整流程

项目内提供脚本模拟一次完整用餐流程（IDLE→SERVING→EATING→IDLE）：

1. 先启动后端：`npm start`。
2. 绑定设备（若尚未绑定）：
   ```bash
   curl -X POST http://localhost:3000/api/v1/devices/bindings \
     -H "Content-Type: application/json" \
     -d '{"device_id":"aa:bb:cc","user_id":1}'
   ```
3. 执行脚本（在项目根目录）：
   ```bash
   bash scripts/test_telemetry_flow.sh
   ```
4. 脚本会连续发送 7 次遥测请求，结束后可在数据库中查看 `Lunchbox_Meals`、`Meal_Curve_Data`、`Meal_Records` 验证。

## 七、常用 API 与操作速查

以下为常用接口，便于调试或与第三方集成。

| 用途 | 方法 | URL / 说明 |
|------|------|------------|
| 健康检查 | GET | `/health` |
| 手动触发日汇总 | POST | `/api/diet/summary/run`，Body 可选 `{"date":"2024-03-08"}` |
| 图表数据 | GET | `/api/diet/statistics/charts?user_id=1&start_date=2024-03-01&end_date=2024-03-08&chart_type=daily_trend`，`chart_type` 可选：`daily_trend`、`weekly_comparison`、`waste_analysis`、`speed_analysis`、`nutrition_pie` |
| 生成 AI 报告 | POST | `/api/diet/analysis/generate`，Body：`{"user_id":1,"date":"2024-03-08","report_type":"daily"}` 或 `"weekly"`，可选 `"force_fallback":true` |
| 查询已有报告 | GET | `/api/diet/analysis/report?user_id=1&date=2024-03-08` |
| 个性化建议 | GET | `/api/diet/recommendations?user_id=1` |
| 设备绑定列表 | GET | `/api/v1/devices/bindings?user_id=1` |
| 绑定设备 | POST | `/api/v1/devices/bindings`，Body：`{"device_id":"xx","user_id":1}` |
| 解绑设备 | DELETE | `/api/v1/devices/bindings/{device_id}` |
| 硬件遥测 | POST | `/api/v1/hardware/telemetry`，Body：`{"device_id":"xx","timestamp":...,"weights":{...}}` |

## 八、常见问题

**Q：前端图表/报告/建议一直加载失败？**  
- 确认后端已启动（`npm start`），且端口为 3000（或与前端代理一致）。  
- 浏览器控制台看是否有跨域或 404，前端开发模式下通过 Vite 代理访问 `/api`，一般不需单独配 CORS。

**Q：生成报告提示没有数据或报错？**  
- 日报需要该日期已有日汇总数据（`Daily_Diet_Summary`）。可先执行一次手动汇总：`POST /api/diet/summary/run`，Body `{"date":"该日期"}`。  
- 周报的 `date` 为周结束日，会聚合该日往前 7 天数据。

**Q：未配置 OpenAI 会怎样？**  
- 不配置 `OPENAI_API_KEY` 时，生成报告会自动使用规则引擎生成降级报告，功能仍可用。

**Q：设备解绑后，之前的用餐记录会删除吗？**  
- 解绑只解除设备与用户的绑定关系，已写入的 `Meal_Records`、`Lunchbox_Meals` 等数据不会自动删除。

**Q：如何确认每日 0 点汇总是否执行？**  
- 若使用 `npm run cron`，可查看该进程日志。也可在次日查看 `Daily_Diet_Summary` 表是否有前一天日期的记录。
